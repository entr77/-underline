## 1. 소셜 독서 앱 기획 원칙 (밑줄 특화)

### 유틸리티(기록) vs 소셜(공유) 기능 균형 원칙

밑줄 서비스는 "기록 도구"이자 "소셜 공간"이다. 둘 중 하나만 강화하면 서비스 정체성이 무너진다.

**균형 원칙**:
- 기록 기능(유틸리티)이 소셜 동기를 만들고, 소셜 기능이 기록 빈도를 높이는 순환 구조를 유지한다.
- 기록하지 않으면 공유할 것이 없다. 따라서 기록 UX의 마찰 최소화가 항상 소셜 기능보다 선행한다.
- 소셜 기능은 기록을 더 의미있게 만드는 방향으로만 추가한다. 기록과 무관한 순수 소셜 기능(DM, 그룹채팅 등)은 이 서비스의 범위 밖이다.

**판단 기준 — 기능 제안이 들어올 때**:

| 판단 질문 | YES | NO |
|-----------|-----|-----|
| 이 기능이 없으면 밑줄을 더 자주 저장하는가? | 유틸리티 강화 | 소셜 강화 쪽인지 재확인 |
| 이 기능이 없으면 밑줄을 남들과 나누기 어려운가? | 소셜 강화 | 필요성 재검토 |
| 두 질문 모두 NO라면 | P2 이하로 분류 또는 기각 |

**균형 가이드라인**:
- 로드맵에서 유틸리티:소셜 기능 비율을 6:4 내외로 유지한다.
- 연속으로 소셜 기능만 2개 이상 추가할 때 반드시 기록 마찰 제거 기능을 병행 검토한다.
- 연속으로 유틸리티 기능만 3개 이상 추가할 때 소셜 디스커버리 강화 기능을 검토한다.

---

### 신규 기능 추가 통과 기준

신규 기능은 아래 두 질문 중 하나에 명확하게 YES여야 한다.

> **"이 기능을 추가하면, 유저가 더 잘 기록하거나 더 많이 공유하게 되는가?"**

| 통과 기준 | 예시 기능 |
|-----------|-----------|
| 더 잘 기록 — 저장 단계의 마찰이 줄거나, 기록의 질이 높아진다 | OCR 정확도 개선, 책 자동 인식 성공률 향상, 밑줄 일괄 편집 |
| 더 많이 공유 — 밑줄을 외부 채널로 내보내거나, 앱 내에서 다른 독자에게 발견된다 | 공유 카드 개선, 팔로우 기능, 책 히트맵 전면 노출 |
| 두 기준 모두 해당 — 가장 높은 우선순위 | 공유 카드(기록 결과를 공유 콘텐츠로 변환) |

**자동 기각 기준** (두 질문 모두 NO인 경우):
- 독서량/완독 통계 대시보드 — 생산성 앱 뉘앙스, 기록/공유 동기 없음
- 독서 타이머 — 유틸리티처럼 보이나 밑줄 기록과 무관
- 도서관/서점 찾기 — 외부 서비스 영역

### 독서 앱 핵심 리텐션 지표

| 지표 | 정의 | 현재 단계 목표 | 성숙 단계 목표 |
|------|------|--------------|--------------|
| **DAU/MAU** | 일간 활성 유저 / 월간 활성 유저 비율 | 15% (초기 코어 유저 기반 형성) | 25% (습관적 사용 달성) |
| **책당 밑줄 수** | 유저가 같은 책에서 저장하는 평균 밑줄 개수 | 3개 이상 | 7개 이상 |
| **공유율** | 저장한 밑줄 중 외부 SNS로 내보낸 비율 | 10% | 20% |
| **D7 리텐션** | 가입 7일 후 재방문 비율 | 30% | 45% |
| **밑줄/유저/월** | 월간 1인당 저장 밑줄 수 | 5개 이상 | 15개 이상 |
| **팔로우 전환율** | 좋아요 누른 후 해당 유저 프로필 방문 비율 | 8% | 15% |

> 측정은 Supabase Analytics 또는 별도 이벤트 트래킹(Mixpanel/PostHog) 연동 후 가능. 현 단계에서는 Supabase 쿼리로 주요 지표를 직접 집계한다.

## 2. 한국 독서 시장 기획 인사이트

### 텍스트힙 트렌드와 밑줄 포지셔닝 전략

**텍스트힙이란**: 숏폼·인스타그램 피로에 대한 반작용으로 긴 텍스트, 종이책, 아날로그 독서가 Z세대 사이에서 "힙한 행위"로 재정의된 현상 (2024년 국내 대표 트렌드).

**밑줄 포지셔닝 전략**:
- 밑줄 서비스는 "독서 기록 앱"이 아니라 "당신의 독서 감수성을 보여주는 곳"으로 포지셔닝한다.
- 유저가 밑줄을 저장하는 행위 자체가 "나는 이런 문장에 멈추는 사람"이라는 정체성 표현임을 기획의 전제로 삼는다.
- 기능 기획 시 "이 기능이 유저의 독서 정체성 표현을 돕는가?"를 부가 검토 질문으로 사용한다.
- 아날로그(손밑줄 사진) + 디지털(OCR 텍스트화 + 공유 카드)의 두 세계를 잇는 역할을 명시적으로 유지한다.

**피해야 할 포지셔닝**:
- 독서량/완독 목표 관리 (자기계발 앱 오인)
- AI 추천 강조 (기술 중심 서비스 오인)
- "더 효율적으로 읽기" 메시지 (텍스트힙 감성 훼손)

### 북스타그램/BookTok 유저가 원하는 기능 우선순위 Top 5

마케팅 리서치 데이터(공유 동기 분석 + 인기 콘텐츠 패턴)를 기반으로 도출.

| 순위 | 원하는 것 | 근거 | 밑줄 대응 기능 |
|------|-----------|------|--------------|
| 1 | 손밑줄 사진을 예쁘게 SNS에 공유 | 손밑줄 실물 사진이 진정성 최강 반응 | 공유 카드 `photo` 테마 (구현 완료) |
| 2 | "나와 같은 문장에 멈춘 사람" 발견 | 공명 감정(68%)이 최대 공유 동기 | 공감자 프로파일(구현 완료), 팔로우(P1 잔여) |
| 3 | 책 정보 자동 입력 (수동 타이핑 없이) | 마찰 최소화 = 기록 빈도 직접 영향 | 책 자동 인식 (구현 완료) |
| 4 | "이 책에서 독자들이 많이 멈춘 문장" 보기 | 사회적 검증 + 책 추천 동기 | `/book/[id]` 히트맵 (구현, 전면화 필요) |
| 5 | 특정 독자 구독/팔로우 | 비슷한 취향 발견 → 연결 | 팔로우 DB 구현 (P1 잔여) |

### 출판사/서점 파트너십 연동 기획 아이디어

> 현재 단계(P2)에서는 직접 구현 없이 아이디어 레벨로 유지. MVP 유저 확보 후 검토.

| 파트너십 유형 | 기획 아이디어 | 기대 효과 |
|-------------|-------------|---------|
| **출판사** | "이 책 독자들의 밑줄 TOP 10" API 제공 → 출판사 마케팅 소재로 활용 | 출판사 자발적 바이럴 유도 |
| **독립서점** | 서점별 큐레이션 계정 — 서점이 추천하는 밑줄 컬렉션 | 오프라인-온라인 연결, 로컬 커뮤니티 |
| **도서관** | 책 ISBN 연동 — "이 도서관 책의 밑줄 모음" (익명 공개 밑줄) | B2B2C 채널 확보 |
| **문화이벤트** | 한강 노벨상처럼 문화적 모멘텀 시기 — 해당 책 밑줄 특집 큐레이션 | 미디어 노출 기회 |

**전제 조건**: 파트너십 기능 진입 전에 MAU 5,000명 이상, 밑줄 누적 50,000개 이상 확보 필요. 그 전에는 오가닉 성장에 집중.

## 3. 기능 우선순위 결정 프레임워크

### P0/P1/P2 판단 기준표

#### P0 — 즉시 구현 (서비스 핵심 가치 직결)

| 카테고리 | 기준 | 예시 |
|----------|------|------|
| 기록 강화 | 밑줄 저장 플로우의 직접 마찰 | OCR 오류 수정 UX, 책 인식 실패 시 수동 검색 |
| 소셜 강화 | 최소 소셜 루프 유지 | 피드 표시, 좋아요, 마이페이지 |
| 리텐션 강화 | 재방문 없으면 서비스 미완성 | 로그인 상태 유지, 알림 기반 재방문 최소 1개 |

**P0 통과 요건**: 이 기능 없이 "밑줄 저장 → 피드 공유 → 다른 독자 발견" 기본 루프가 동작하지 않는가? YES이면 P0.

#### P1 — 다음 이터레이션 (핵심 루프 강화)

| 카테고리 | 기준 | 예시 |
|----------|------|------|
| 기록 강화 | 저장 품질/속도 향상, 기록 재활용 | 밑줄 편집, "오늘의 내 밑줄" 복습 알림 |
| 소셜 강화 | 발견 + 연결 레이어 추가 | 팔로우/팔로잉, 책 히트맵 전면화 |
| 리텐션 강화 | D7 리텐션 개선에 직접 기여 | 푸시 알림, 온보딩 개선 |

**P1 통과 요건**: P0 루프가 안정적으로 동작하는 상태에서, DAU/D7 리텐션/공유율 중 하나를 10% 이상 개선할 것으로 기대되는가?

#### P2 — 백로그 (장기 차별화)

| 카테고리 | 기준 | 예시 |
|----------|------|------|
| 기록 강화 | 헤비유저를 위한 고급 기능 | 밑줄 태그/컬렉션, 검색 |
| 소셜 강화 | 커뮤니티 레이어 | 북클럽, 댓글, DM |
| 리텐션 강화 | 알고리즘 기반 개인화 | 추천 엔진, 스페이스드 리피티션 |

**P2 통과 요건**: MAU 10,000명 이상 또는 핵심 지표가 목표치에 도달한 이후 검토.

---

### 현재 백로그에서 다음에 해야 할 기능과 그 근거

**즉시 착수 권장: 팔로우 DB 구현 (P1 잔여)**

- **현황**: 팔로우 UI는 구현되어 있으나 DB 미구현 상태 (PRD 명시)
- **근거**: 소셜 루프의 핵심 연결 고리. "같은 문장에 멈춘 독자"를 발견한 후 팔로우가 안 되면 재방문 동기가 소멸. D7 리텐션에 직접 영향.
- **예상 공수**: 마이그레이션(follows 테이블) + RLS + API + UI 연결 = 개발 1~2일

**두 번째: 책 히트맵 전면화**

- **현황**: `/book/[id]` 히트맵 구현되어 있으나 피드에서 진입 동선 약함
- **근거**: 마케터 인사이트 — "이 책에서 가장 많이 밑줄 친 문장" 노출이 신규 유저 발견 + 공유 유도 최강 훅. `/books` 피드 카드에 "독자 N명이 멈춘 문장" 스니펫 추가.
- **예상 공수**: 쿼리 + UI 변경 = 개발 0.5일

**세 번째: "오늘의 내 밑줄" 복습 알림 (P1)**

- **현황**: 미구현
- **근거**: Readwise의 핵심 리텐션 장치를 간소화. 저장한 밑줄을 랜덤 하나씩 알림으로 — D7 이후 리텐션 하락 방어. 구현 복잡도 낮고 리텐션 임팩트 높음.
- **예상 공수**: 푸시 알림 인프라(Web Push) + 스케줄러 = 개발 2~3일

## 4. 유저 스토리 품질 기준 (밑줄 특화)

### 좋은 유저 스토리 예시 3개

**예시 1 — 기록 강화 (팔로우 기능)**

```
As a 독서 피드를 탐색하다 마음에 드는 독자를 발견한 사용자,
I want to 해당 독자를 팔로우해서 그 사람의 밑줄만 골라 볼 수 있도록,
So that 취향이 비슷한 독자의 새 밑줄을 놓치지 않고 발견할 수 있다.

수용 기준:
- [ ] 프로필 페이지에서 팔로우 버튼을 탭하면 팔로우 상태로 즉시 전환된다 (optimistic update)
- [ ] 팔로우한 독자의 새 밑줄이 홈 피드 상단에 우선 노출된다
- [ ] 팔로우 수 / 팔로워 수가 프로필에 표시된다
- [ ] 본인 계정은 팔로우 버튼이 표시되지 않는다
- [ ] 언팔로우 시 피드에서 즉시 제외된다 (새로고침 없이)
- [ ] RLS: 팔로우 데이터는 본인과 팔로잉 대상만 조회 가능

제외 범위:
- 팔로우 알림 (별도 스토리)
- 맞팔 개념 / 비공개 계정
```

**예시 2 — 소셜 강화 (책 히트맵 전면화)**

```
As a 어떤 책을 읽을지 고민 중인 독자,
I want to 책 피드(/books)에서 "이 책에서 독자들이 가장 많이 멈춘 문장"을 미리 보고 싶다,
So that 책을 구매하기 전에 그 책의 감성과 깊이를 먼저 느낄 수 있다.

수용 기준:
- [ ] /books 피드의 각 북카드에 좋아요 기준 상위 1개 밑줄 문장이 스니펫으로 표시된다
- [ ] 스니펫은 최대 2줄(약 60자)로 줄임표 처리된다
- [ ] 스니펫 탭 시 해당 밑줄 상세(/underline/[id])로 이동한다
- [ ] "N명이 밑줄 친 문장" 카운트가 함께 표시된다
- [ ] 밑줄이 0개인 책은 스니펫 없이 표지+제목만 표시된다

제외 범위:
- 페이지 히트맵 시각화 (책 상세 페이지에만 유지)
- AI 요약 / 자동 추천 문장
```

**예시 3 — 리텐션 강화 (복습 알림)**

```
As a 예전에 저장한 밑줄들을 잊고 지내는 활성 유저,
I want to 매일 하나씩 내 과거 밑줄을 알림으로 받아보고 싶다,
So that 좋아했던 문장을 다시 만나고 앱에 자연스럽게 재방문하게 된다.

수용 기준:
- [ ] 설정에서 복습 알림 ON/OFF + 시간대 설정 가능 (기본 오전 8시)
- [ ] 저장한 밑줄이 3개 이상인 유저에게만 알림이 발송된다
- [ ] 알림 탭 시 해당 밑줄 상세 페이지로 이동한다
- [ ] 같은 밑줄이 7일 내에 중복 발송되지 않는다
- [ ] 알림 거부 시 브라우저 권한 재요청 없이 설정에서만 관리된다

제외 범위:
- 스페이스드 리피티션 알고리즘 (랜덤 발송으로 시작)
- 읽음 확인 / 퀴즈 형식
```

## 5. 기획 안티패턴 (밑줄에서 피해야 할 것)

### 독서량 게이미피케이션 지양 이유

**무엇인가**: 완독 뱃지, 월간 독서 목표, 독서 스트릭, 연속 기록 일수 등 독서 행위를 수치화/경쟁화하는 기능.

**왜 피해야 하는가**:
1. **정체성 충돌**: 밑줄 서비스의 핵심 정체성은 "감수성 표현"이다. 독서량 게이미피케이션은 서비스를 "자기계발 앱"으로 오인하게 만들어 타겟 유저(북스타그램/BookTok 독자)를 이탈시킨다.
2. **행동 왜곡**: 뱃지와 스트릭은 "빨리 읽기"를 유도해 실제로 밑줄을 남길 여유를 감소시킨다. 저장되는 밑줄 수와 역상관 관계가 생긴다.
3. **마케팅 언어 금지 위반**: "독서량 관리" 뉘앙스는 마케팅 리서치에서 명시적으로 금지한 표현 방향이다.
4. **텍스트힙 역행**: 독서를 목표로 측정하는 방식은 "힙한 아날로그 행위"라는 포지셔닝과 정면 충돌한다.

**허용 범위**: 내 밑줄 총 개수 표시(마이페이지), 책당 밑줄 수 표시(책 상세) — 이것은 기록의 결과물 확인이지 게이미피케이션이 아니다.

---

### 조기 추천 알고리즘 도입 위험성

**무엇인가**: 유저 데이터 기반의 책/밑줄 추천 엔진을 유저 수가 적은 초기에 도입.

**왜 위험한가**:
1. **데이터 부족**: 추천 알고리즘은 유저 1인당 최소 20개 이상의 밑줄 + 좋아요 데이터가 있어야 의미있는 추천을 생성한다. 초기에는 "인기 책 추천"과 다를 바 없어 유저에게 가치가 없다.
2. **콜드 스타트 문제**: 신규 유저의 첫 경험에 "추천 없음" 화면이 뜨면 오히려 이탈을 유발한다.
3. **엔지니어링 비용**: 추천 인프라는 구축 비용이 높다. 초기에는 같은 리소스로 기록/공유 마찰 제거에 쓰는 것이 ROI가 압도적으로 높다.
4. **대안 존재**: 추천 없이도 "이 책의 독자 N명이 멈춘 문장" + "나와 같은 밑줄을 좋아한 독자" 로직으로 소셜 디스커버리를 구현할 수 있다. 알고리즘 불필요.

**도입 기준**: MAU 10,000명 + 밑줄 누적 100,000개 이상 확보 후 P2로 진행.

---

### 복잡한 태그/분류 시스템 경계 기준

**무엇인가**: 밑줄에 다중 태그 추가, 폴더/컬렉션 분류, 취향 카테고리 트리 구성 등.

**왜 경계해야 하는가**:
1. **저장 마찰 증가**: 밑줄 저장 플로우에 태그 선택 단계가 추가되면 저장 완료율이 감소한다. "나중에 태그 달아야지"가 이탈로 이어진다.
2. **비어있는 분류 체계**: 유저가 적을 때 복잡한 분류 시스템은 빈 카테고리만 보여준다. 오히려 서비스가 덜 채워진 것처럼 느끼게 만든다.
3. **취향 표현 방식 왜곡**: 밑줄 자체가 이미 취향의 표현이다. 별도 태그로 재분류하면 유저에게 "두 번 표현"을 요구하는 꼴이 된다.

**허용 경계**:
- 온보딩에서 최대 5개 취향 태그 선택 (프로필 표시용, 밑줄에 직접 붙이지 않음)
- 밑줄 저장 시 자동 추출되는 책 장르 태그 (유저 입력 없음)
- 공개/비공개 토글 — 이것은 분류가 아닌 접근 제어

**금지 경계**:
- 밑줄 저장 시 수동 태그 입력 요구
- 3단계 이상 중첩 분류 (장르 > 서브장르 > 감정태그 등)
- 태그 기반 자동 컬렉션 생성 (초기 단계)
