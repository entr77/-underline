### 단독 처리 vs 복합 처리 판단 기준

```
단독 처리 가능:
  - 기존 컴포넌트 스타일 수정 (designer)
  - 버그 수정 — 로직 변경 없음 (developer)
  - 카피 문구 수정 (copywriter)
  - 법적 문서 검토 (legal)
  - 성능 최적화 — 쿼리 튜닝 (developer)

복합 처리 필요:
  - 신규 기능 추가 (항상 planner → ... → reviewer)
  - DB 스키마 변경 포함 (planner → developer → reviewer)
  - 신규 페이지 생성 (designer → developer)
  - 외부 API 신규 연동 (planner → developer → reviewer)
```

### 복합 요청 처리 표준 순서

**신규 기능 (Full Flow)**
```
planner (유저 스토리 + 수용 기준)
  → designer (화면 설계 + 컴포넌트 스펙)
  → developer (구현 + DB 마이그레이션)
  → reviewer (코드 리뷰 + RLS 검증)
```

**DB 스키마 변경 없는 기능 개선**
```
designer (변경 스펙 확정)
  → developer (구현)
  → reviewer (코드 리뷰)
```

**버그 수정**
```
developer (근본 원인 파악 + 전체 범위 수정)
  → reviewer (회귀 가능성 점검)
```

**보안/법적 이슈**
```
legal 또는 reviewer (이슈 확인 + 범위 정의)
  → developer (수정)
  → reviewer (재검증)
```

### 에이전트 결과물 품질 검증 기준 (DoD 요약)

각 에이전트 결과물을 다음 에이전트에게 넘기기 전, PM이 체크한다.

| 에이전트 | 넘기기 전 체크 포인트 |
|---------|---------------------|
| planner | 수용 기준(Acceptance Criteria) 명시되어 있는가? Confluence PRD 업데이트했는가? |
| designer | 모바일(375px) 기준 설계인가? 디자인 토큰(`var(--color-*)`) 사용하는가? Confluence UI/UX 업데이트했는가? |
| developer | `tsc --noEmit` 통과하는가? `.env.local` 커밋 없는가? 에러 처리에 `<Alert>` 사용하는가? |
| reviewer | 지적 사항에 severity 분류(blocking/major/minor)가 있는가? 해결 방법 명시되어 있는가? |

## 2. 태스크 분해 기준 (밑줄 서비스 특화)

### Epic / Feature / Story / Task 크기 기준

| 단위 | 정의 | 크기 기준 | 예시 |
|------|------|-----------|------|
| **Epic** | 서비스 기능 도메인 | 2주 이상 | 팔로우 소셜 기능 전체 |
| **Feature** | 유저가 사용 가능한 단위 기능 | 2~5일 | 팔로우/언팔로우 기능 |
| **Story** | 유저 관점의 단위 작업 | 0.5~1일 | "독자 프로필에서 팔로우 버튼을 눌러 팔로우할 수 있다" |
| **Task** | 개발자/에이전트 단위 작업 | **4시간 이내** | `follows` 테이블 마이그레이션 SQL 작성 |

### Task 4시간 원자성 기준

Task가 4시간을 초과할 것으로 예상되면 즉시 분할한다.

**분할 신호:**
- 파일 변경이 5개 이상 필요할 때
- 새로운 DB 테이블 + API + UI가 동시에 필요할 때
- 외부 API 연동이 포함될 때 (연동 자체를 별도 Task로)
- 테스트/검증이 구현과 분리 가능할 때

**분할 방법:**
```
큰 Task: "팔로우 기능 구현"
  → Task 1: follows 테이블 마이그레이션 + RLS (1h)
  → Task 2: /api/follows POST/DELETE 엔드포인트 (2h)
  → Task 3: FollowButton 컴포넌트 + optimistic update (2h)
  → Task 4: /profile 페이지 팔로우 수 표시 (1h)
```

## 3. 스프린트 계획 (인디 1인 서비스 특화)

### 1주 스프린트 구조

인디 1인 개발 특성상 PM/planner/designer/developer를 혼자 순환한다. 스프린트는 7일 단위.

| 요일 | 활동 | 에이전트 역할 |
|------|------|-------------|
| 월 (Plan) | 스프린트 목표 확정, 백로그 정제, 태스크 분해 | PM |
| 화~목 (Build) | 코어 구현 집중 | developer |
| 금 (Review) | 코드 리뷰, 버그 수정, 테스트 | reviewer |
| 토 (Polish) | UI 개선, 성능 최적화, 문서화 | designer / PM |
| 일 (Rest) | 스프린트 회고, 다음 스프린트 초안 | PM |

**1주 스프린트 용량 기준:**
- 개발 가능 시간: ~20시간 (파트타임 기준)
- 코어 Feature 1개 + 버그 수정 2~3개 + 기술 부채 1개

### P0 / P1 / P2 + 의존성 기반 우선순위 결정

**우선순위 점수 공식 (간소화):**
```
점수 = (비즈니스 임팩트 × 2) + (유저 요구도) - (개발 복잡도) - (의존성 미해결 수 × 3)
```

| 기준 | P0 | P1 | P2 |
|------|----|----|-----|
| 비즈니스 임팩트 | 서비스 핵심 동작에 필수 | 유저 경험 유의미한 개선 | 장기 성장 기반 |
| 유저 노출 빈도 | 매 세션 | 주 1회 이상 | 선택적 |
| 의존성 | 다른 P0 없음 | P0 완료 후 가능 | P1 완료 후 가능 |
| 미구현 시 영향 | 서비스 불가 | 불편함 | 기회 손실 |

**의존성 기반 재우선순위:**
```
팔로우 기능 (P1):
  - follows 테이블 없으면 팔로잉 피드 불가 → DB 먼저
  - 팔로잉 피드 없으면 팔로우 동기 약함 → 피드와 동시 출시

추천 엔진 (P2):
  - 태그 시스템 없으면 추천 품질 낮음 → 태그 먼저
  - 유저 수 < 100이면 추천 데이터 부족 → 그로스 후 개발
```

### 블로커 처리 프로세스

```
블로커 발생
  ↓
1. 즉시 가시화 — 현재 스프린트 계획에 [BLOCKER] 태그 추가
2. 분류:
   - 기술 블로커 (API 한도, 라이브러리 버그) → 우회책 찾기 or 범위 축소
   - 외부 블로커 (Supabase 다운, API 키 만료) → 대기 중 다른 Task 진행
   - 의사결정 블로커 (기획 미확정) → planner에게 즉시 위임
3. 해결 기한 설정 (최대 1일)
4. 기한 내 미해결 시 → 해당 Story를 다음 스프린트로 이동
5. 해결 후 Confluence에 결과 기록
```

**자동 블로커 조건:**
- 외부 API 연속 3회 실패
- `tsc --noEmit` 오류 미해결 상태로 커밋 시도
- Supabase 마이그레이션 미실행 상태에서 코드 배포

## 4. Confluence 문서화 정책

### 문서화 트리거 — 반드시 문서화해야 하는 결정

| 결정 유형 | 문서화 타이밍 | 대상 페이지 |
|----------|-------------|-----------|
| 신규 기능 기획 확정 | 개발 시작 전 | PRD (164058) |
| 화면 설계 확정 | 개발 시작 전 | UI/UX 설계 (360619) |
| DB 스키마 변경 | 마이그레이션 SQL 작성과 동시에 | DB 스키마 (589825) |
| 외부 API 신규 도입 | 첫 커밋 전 | 기술 스택 & 아키텍처 (622593) |
| 기술 스택 변경 (라이브러리 추가/제거 포함) | 결정 직후 | 기술 스택 & 아키텍처 (622593) |
| 디자인 토큰 변경 | CSS 변수 수정과 동시에 | UI/UX 설계 (360619) |
| 마케팅 인사이트 / 그로스 전략 | 분석 완료 후 | 마케팅 리서치 (1245185) |
| 보안 정책 변경 (RLS, 인증 흐름) | 코드 머지 전 | DB 스키마 또는 기술 스택 |
| 기능 중단(Sunset) 결정 | 결정 직후 | PRD (164058) |

### 문서화 면제 조건 (문서화 불필요)

- 버그 수정 (기능 변경 없는 로직 수정)
- 오타/문구 수정
- 성능 최적화 (API/DB 변경 없음)
- 린트/포매터 설정 변경

### 페이지 업데이트 vs 새 페이지 생성 기준

| 기준 | 기존 페이지 업데이트 | 새 페이지 생성 |
|------|-------------------|-------------|
| 내용 성격 | 기존 섹션의 내용이 변경됨 | 기존 어느 페이지에도 속하지 않는 독립 주제 |
| 크기 | 변경 분량 < 전체 페이지 50% | 새 내용이 기존 페이지 분량을 초과 |
| 예시 | PRD의 기능 상태 업데이트 | 새 에이전트 정책 문서 |
| 예시 | UI/UX의 컴포넌트 스펙 추가 | 배포 런북, 인시던트 기록 |

### 코드 변경과 문서 동기화 체크포인트

**커밋 전 자가 체크리스트:**

```
[ ] DB 테이블/컬럼 추가/변경 → DB 스키마 페이지 업데이트했는가?
[ ] 새 환경변수 추가 → 기술 스택 & 아키텍처에 목록 업데이트했는가?
```

## 5. 각 에이전트 결과물 DoD (Definition of Done)

### Planner DoD

다음을 모두 충족해야 planner 작업이 완료된 것으로 간주한다.

- [ ] **유저 스토리 형식**: "나는 [역할]로서 [기능]을 [이유] 때문에 원한다"
- [ ] **수용 기준 명시**: 각 Story에 체크 가능한 수용 기준 3개 이상
- [ ] **범위 경계 명확**: 이번 이터레이션에 포함/제외되는 것이 명확히 구분됨
- [ ] **의존성 명시**: 선행 조건이 있으면 명시 (예: "팔로우 테이블 없으면 불가")
- [ ] **우선순위 지정**: P0/P1/P2 태그 부여
- [ ] **Confluence PRD 업데이트**: 기능 상태, 스펙 변경 반영 완료
- [ ] **Designer에게 인수인계**: 화면 설계에 필요한 데이터 플로우/상태 명시

### Designer DoD

- [ ] **Mobile First**: 375px 기준 와이어프레임/스펙 작성 완료
- [ ] **디자인 토큰 준수**: 색상은 `var(--color-*)`, 폰트는 font-serif/sans 사용
- [ ] **컴포넌트 재사용**: 기존 컴포넌트(Alert, ProfileChip, TagBadge 등) 활용 명시
- [ ] **Alert 규칙**: 에러/경고/성공 메시지는 `<Alert>` 사용 명시
- [ ] **반응형 명세**: 768px(태블릿), 1280px(데스크탑) 확장 스펙 포함
- [ ] **인터랙션 명세**: 로딩 상태, 에러 상태, 빈 상태(empty state) 설계 포함
- [ ] **Confluence UI/UX 업데이트**: 컴포넌트 표, 페이지 구조 반영 완료
- [ ] **Developer에게 인수인계**: 컴포넌트명, props 타입, 레이아웃 치수 명시

### Developer DoD

- [ ] **타입 체크 통과**: `tsc --noEmit` 오류 없음
- [ ] **빌드 통과**: `npm run build` 성공 (Vercel 배포 기준)
- [ ] **Server Component 기본**: `"use client"` 최소화, 인터랙션 필요한 경우만
- [ ] **에러 처리**: 모든 에러 메시지는 `<Alert variant="error">` 사용
- [ ] **보안 준수**: service role key 서버에서만, RLS 정책 적용 확인
- [ ] **환경변수 보호**: `.env.local` 커밋 없음, 새 환경변수는 Vercel에도 추가
- [ ] **API 라우트 위치**: 서버 로직은 `/src/app/api/` 하위
- [ ] **기존 패턴 일관성**: 같은 기능의 기존 구현 패턴과 동일하게 작성
- [ ] **DB 마이그레이션 파일**: `supabase/migrations/` 에 SQL 파일 추가 (스키마 변경 시)
- [ ] **Reviewer에게 인수인계**: 변경 파일 목록, 테스트 방법, 주의사항 명시

### Reviewer 통과 기준

리뷰어는 아래 기준으로 severity를 분류한다.

| Severity | 정의 | 처리 방법 |
|----------|------|----------|
| **blocking** | 머지 불가 — 보안 취약점, 데이터 손실 위험, 빌드 오류 | 반드시 수정 후 재리뷰 |
| **major** | 머지 전 수정 권고 — 잘못된 RLS, N+1 쿼리, 메모리 누수 | 수정 후 머지 가능 |
| **minor** | 선택적 개선 — 코드 스타일, 주석 부재, 네이밍 | 다음 PR에서 처리 가능 |

**Reviewer 통과 체크리스트:**

- [ ] **RLS 정책 검증**: 새 테이블/컬럼에 적절한 RLS 설정
- [ ] **인증 흐름 검증**: 인증 필요 페이지/API에 세션 체크 존재
- [ ] **SQL Injection 가능성**: 동적 쿼리 사용 시 파라미터 바인딩 확인
- [ ] **서버/클라이언트 분리**: 민감 키가 클라이언트에 노출되지 않음
- [ ] **에러 처리 완결성**: try/catch 없는 async 코드 없음
- [ ] **TypeScript 타입 안전성**: `any` 사용 최소화, 타입 단언(`as`) 남용 없음
- [ ] **severity 없음**: blocking/major 이슈 0건이어야 머지 승인

## 6. 리스크 관리 & 에스컬레이션

### 즉시 PM 에스컬레이션이 필요한 케이스

| 케이스 | 에스컬레이션 기준 | PM 액션 |
|--------|----------------|---------|
| 외부 API 장애 | 서비스 핵심 기능 3분 이상 중단 | 대응 계획 발동, 사용자 공지 여부 결정 |
| 데이터 손실 위험 감지 | reviewer가 blocking 이슈 발견 | 배포 즉시 중단, developer에게 핫픽스 위임 |
| 저작권/개인정보 이슈 | legal이 리스크 플래그 | 기능 일시 중단 후 legal 처리 완료 후 재개 |
| 스프린트 목표 50% 미달 예상 | 수요일 기준 | 범위 조정 (scope down) 결정 |
| 기획 변경 요청 (개발 중) | 현재 스프린트 Story 영향 | 즉시 스프린트에서 제외 후 다음 스프린트 검토 |
| Supabase 무료 티어 한도 근접 | DB 용량 80% 도달 | 정리/업그레이드 플랜 결정 |

### Legal 플래그 발생 시 처리 프로세스

```
Legal 플래그 발생
  ↓
1. 해당 기능/코드 즉시 비활성화 (feature flag 또는 코드 롤백)
2. Legal 에이전트에게 전문 검토 위임
   - 저작권: 책 텍스트 저장/표시 범위 검토
   - 개인정보: 사용자 사진/데이터 처리 방식 검토
   - 약관: 이용약관·개인정보처리방침 업데이트 필요 여부
3. Legal 판단 결과:
   - 허용: 기능 재활성화, 필요시 면책 문구 추가
   - 조건부 허용: 조건 충족 후 재활성화
   - 불허: 기능 제거, PRD에서 해당 항목 삭제
4. 결과를 Confluence PRD에 법적 검토 내역으로 기록
```

**밑줄 서비스 주요 Legal 리스크 목록:**

| 리스크 | 현황 | 대응 방안 |
|--------|------|----------|
| 책 본문 문장 저장 | 인용 범위 내 허용 (공정 이용) | 표시 분량 제한, 출처 명시 의무화 |
| 책 표지 이미지 표시 | 카카오 API 제공 이미지 사용 | 카카오 API 약관 준수, 자체 크롤링 금지 |
| 사용자 업로드 사진 저장 | 본인 촬영 이미지 → 문제없음 | 타인 저작물 업로드 금지 약관 명시 |
| Unsplash 이미지 사용 (scene 카드) | Unsplash API 무료 라이선스 | API 약관 준수, 출처 표기 |
| Google/Kakao 계정 연동 | OAuth 표준 준수 | 각 플랫폼 개발자 정책 갱신 모니터링 |

### 외부 API 장애 시 대응 계획

#### 카카오 도서 API 장애

| 단계 | 조치 |
|------|------|
| 감지 | `/api/books/search` 응답 실패 3회 이상 |
| 즉시 조치 | 책 자동 인식 비활성화, 수동 입력 폴백 활성화 |
| 사용자 안내 | "책 자동 검색이 일시적으로 불가합니다. 직접 입력해주세요." (Alert info) |
| 복구 확인 | 카카오 상태 페이지 확인 + 5분 간격 재시도 |
| 장기 장애(24h+) | 수동 입력을 정식 플로우로 임시 전환, PRD에 기록 |

**폴백 구현 위치:** `src/app/new/` 의 책 검색 스텝 — 카카오 API 실패 시 `BookSearchInput` 컴포넌트가 수동 입력 모드로 전환.

#### Claude Vision API 장애 (OCR / 밑줄 감지)

| 단계 | 조치 |
|------|------|
| 감지 | `/api/vision/analyze` 응답 실패 또는 타임아웃 |
| 즉시 조치 | OCR 결과 없이 수동 입력 모드로 폴백 (`/new` 스텝 2 → 스텝 4 직접 이동) |
| 사용자 안내 | "사진 자동 분석이 일시적으로 불가합니다. 직접 입력해주세요." (Alert warning) |
| 비용 모니터링 | Claude API 월 사용량 80% 도달 시 PM 에스컬레이션 |
| 장기 장애 | 텍스트 직접 입력 플로우를 기본 모드로 임시 전환 |

**폴백 구현 위치:** `src/lib/vision/orchestrator.ts` — 각 Analyzer 실패 시 빈 결과 반환, UI에서 수동 입력으로 전환.

#### Supabase 장애 (DB / Auth / Storage)

| 장애 유형 | 즉시 조치 | 복구 후 조치 |
|----------|----------|------------|
| Auth 서비스 다운 | 로그인 불가 안내, 읽기 전용 모드(비로그인 피드) 유지 | 세션 만료 사용자 재로그인 안내 |
