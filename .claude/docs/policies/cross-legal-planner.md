## 현재 구현 기능 법적 리스크 평가

PRD P0/P1 전체 기능을 기준으로 평가한다.

| 기능 | 리스크 레벨 | 현재 대응 | 추가 조치 |
|------|-------------|-----------|-----------|
| **회원가입 (이메일)** | Low | Supabase Auth, HTTPS | 개인정보처리방침 게시 전 서비스 런칭 금지 (현재 미게시 — High Risk) |
| **회원가입 (Google OAuth)** | Low-Medium | Google OAuth 표준 플로우 | 처리방침에 Google 제3자 제공 현황 기재 필수 |
| **밑줄 추가 — 사진 업로드** | High | 미확인 | EXIF 위치정보 제거 확인 즉시 필요. `/api/vision/analyze` 경로에서 sharp 등으로 EXIF 스트립 처리 여부 점검 |
| **밑줄 추가 — OCR (Claude Vision)** | Medium | 서버에서만 API 호출 | 처리방침에 Claude API 이미지 처리 위탁 현황 기재. 얼굴 포함 이미지 생체정보 리스크 명시 |
| **밑줄 추가 — 책 문장 저장** | Medium | 밑줄 감지 문장만 저장 (highlight-analyzer) | 저장 시 책 제목·저자·출판사 반드시 함께 저장 확인. 현재 DB 구조(books 테이블 → underlines.book_id)는 적합 |
| **책 자동 인식 (카카오 도서 API)** | Low-Medium | 서버에서만 호출 (`/api/books/search`) | 카카오 API 검색 결과 캐싱 유효기간 24시간 이내 설정. 책 표지 이미지는 카카오 URL만 참조, Supabase Storage 재저장 금지 |
| **페이지 번호 감지** | Low | OCR 보조 기능 수준 | 별도 법적 조치 불필요 |
| **공개 피드 (/feed)** | Low-Medium | 공개 게시 구조 | 각 밑줄 카드에 책 제목·저자 표시 확인. 저작권 침해 신고 채널 개설 전까지 공개 서비스 리스크 존재 |
| **공개/비공개 선택** | Low | 구현 완료 | 비공개 밑줄은 서비스 홍보 목적 사용 제외 조항 이용약관에 추가 권장 |
| **좋아요** | Low | optimistic update, RLS | 좋아요 기록은 계약 이행 근거. 탈퇴 후 즉시 삭제 정책 처리방침에 명시 |
| **마이페이지 (/profile/[username])** | Low | RLS로 본인 데이터 보호 | 프로필 사진 수집 시 별도 동의 근거 처리방침에 명시 |
| **책별 밑줄 모음 + 히트맵 (/book/[id])** | Medium | 집적 구조 | 동일 책 전체 밑줄 집적은 사실상 책 내용 재구성 리스크. 히트맵 전면 노출 시 "가장 많이 밑줄 친 문장" 노출이 원저작물 홍보로 해석될 수 있어 출처 표시 강화 필요 |
| **공감자 프로파일** | Low | 유저 간 익명 연결 수준 | 공감자 닉네임/아바타 노출은 공개 프로필 범위 내. 별도 조치 불필요 |
| **책 단위 피드 (/books)** | Low-Medium | 카카오 데이터 표시 | 카카오 API 출처 표시 의무 확인 ("카카오 제공" 또는 로고 표시 검토) |
| **공유 이미지 카드** | Low-Medium | 출처 표시 구조 | 모든 카드에 책 제목·저자 표시 확인. 카드가 외부 SNS로 나갈 때 저작물이 카드의 주인공이 되지 않도록 레이아웃 설계 유지 (현재 테마별 스펙 적합) |

### 즉시 조치 필요 항목 (현재 구현 기능)

1. **EXIF 위치정보 제거 — High Risk**: 이미지 업로드 파이프라인에서 EXIF 스트립 처리 여부 즉시 확인.
2. **개인정보처리방침 게시 — High Risk**: `/privacy` 경로에 미게시 상태. 서비스 공개 전 필수.
3. **이용약관 게시 — High Risk**: `/terms` 경로에 미게시 상태. UGC 라이선스, 저작권 침해 신고 절차, 계정 제재 기준 포함 필수.
4. **저작권 침해 신고 채널 — High Risk**: 이메일 신고 채널(예: copyright@underline.kr) 개설 및 이용약관에 명시.

## 계획된 기능 법적 리스크 사전 검토

### 팔로우 기능 (P1 잔여)

**기획 현황**: UI 구현 완료, DB(follows 테이블) 미구현.

| 리스크 항목 | 세부 내용 | 대응 방향 |
|-------------|-----------|-----------|
| 팔로우 관계 데이터 | 누가 누구를 팔로우하는지가 개인 네트워크 정보에 해당. PIPA 제15조 계약 이행 근거로 처리 가능 | 처리방침에 팔로우 관계 데이터 수집 목적 명시 |
| 팔로워 수 공개 | 팔로워 수는 사회적 관계를 드러내는 정보 — 원칙적으로 유저가 공개 동의한 정보로 처리 가능 | 프로필 공개 범위를 유저가 설정 가능하도록 권장 (현재 미구현, P2 고려) |
| 타인 팔로우 정보 공개 | "내가 팔로우한 사람 목록"을 제3자에게 공개할 경우 개인정보 제3자 제공 해당 가능 | 팔로우 목록은 본인 + 팔로잉 대상만 조회 가능하도록 RLS 설계 (planner-policy.md 수용 기준과 일치) |
| 차단/숨기기 미구현 | 팔로우 허용 시 원치 않는 팔로워로부터 보호 수단 필요 — 정보통신망법 이용자 보호 의무 | 팔로우 기능과 동시에 차단 기능 최소 버전 설계 권장 (계정 공개/비공개 토글 또는 차단) |
| RLS 설계 | follows 테이블 RLS 누락 시 전체 팔로우 관계 노출 | 수용 기준에 "RLS: 팔로우 데이터는 본인과 팔로잉 대상만 조회 가능" 명시됨 — 개발 시 반드시 준수 |

**Legal 검토 결론**: 팔로우 기능 자체는 Low-Medium 리스크. RLS 설계와 차단 기능 최소 버전 병행 구현 권장.

### 추천 엔진 (P2 로드맵)

**기획 현황**: P2 장기 로드맵. MAU 10,000명 + 밑줄 누적 100,000개 이상 확보 후 검토 (planner-policy.md 기준).

| 리스크 항목 | 세부 내용 | 대응 방향 |
|-------------|-----------|-----------|
| 자동화된 결정 고지 의무 | 개인정보보호법 제37조의2 — 추천 로직이 자동화됨을 이용자에게 고지 의무 | 추천 결과 UI에 "AI 추천" 또는 "취향 기반 추천" 라벨 표시 |
| 프로파일링 데이터 명시 | 추천에 사용되는 데이터 항목(밑줄 기록, 좋아요, 독서 이력)을 처리방침에 명시해야 함 | 추천 엔진 개발 전 처리방침 업데이트 필수 |
| 개인화 거부권 | 자동화된 결정 거부권 — 정보주체가 개인화 추천을 끌 수 있는 옵션 제공 필요 | 설정 페이지에 "개인화 추천 끄기" 토글 필수 |
| 독서 이력의 민감성 | 어떤 책을 읽는지는 정치적 성향·종교·건강 등 민감정보를 간접적으로 드러낼 수 있음 | 추천 알고리즘에 민감정보 추론 방지 가드 설계. 처리방침에 민감정보 간접 처리 가능성 명시 |
| 제3자 프로파일 판매 금지 | 추천을 위해 수집된 독서 프로파일을 외부 광고사 등에 제공하는 것은 PIPA 위반 | 추천 기능은 서비스 내부 목적으로만 사용. 외부 제공 금지를 처리방침에 명시 |

**Planner 주의사항**: planner-policy.md의 "조기 추천 알고리즘 도입 위험성" 원칙과 법적 요건이 일치한다. 충분한 데이터 확보 전 도입은 기획 리스크와 법적 리스크가 동시에 높다. P2 진입 기준(MAU 10,000명)을 법적 준비 완료 시점과 함께 체크포인트로 설정할 것.

### 알림 서비스 (P1 — "오늘의 내 밑줄" 복습 알림)

**기획 현황**: 복습 알림 미구현. planner-policy.md에서 P1 세 번째 우선순위로 권장.

| 알림 유형 | 법적 분류 | 동의 필요 여부 | 구현 요건 |
|-----------|-----------|----------------|-----------|
| 복습 알림 ("오늘의 내 밑줄") | 서비스 알림 (비광고) | 별도 수신 동의 불필요 — 서비스 이용 계약 범위 | 설정에서 ON/OFF + 시간대 설정 필수 (planner 수용 기준과 일치) |
| 좋아요 알림 | 서비스 알림 (비광고) | 별도 수신 동의 불필요 | 서비스 이용 계약 범위 |
| 팔로워 알림 | 서비스 알림 (비광고) | 별도 수신 동의 불필요 | 팔로우 기능 구현 후 연동 |
| 신규 도서 추천 알림 | 광고성 정보 | **별도 수신 동의 필수** | DB에 동의 여부 기록 + 수신 거부 처리 이력 6개월 보존 |
| 마케팅 이메일 | 광고성 정보 | **별도 수신 동의 필수** | 수신거부 링크 포함 필수 (정보통신망법 제50조) |

**복습 알림 구현 시 Legal 요건**:
- Web Push 권한 요청 전 사용자에게 알림 목적 고지
- 알림 거부 후 브라우저 권한 재요청 금지 (planner 수용 기준과 일치)
- 발송 이력 관리 — 개인정보이므로 보존 기간 처리방침에 명시

## Planner가 기획 단계에서 Legal에 물어야 할 체크리스트

새 기능 기획 완료 후 개발 시작 전, 아래 질문에 하나라도 해당하면 Legal 검토 요청 필수.

### 데이터 수집 관련

- [ ] 이 기능이 현재 처리방침에 명시되지 않은 새로운 개인정보 항목을 수집하는가?
- [ ] 이 기능이 유저 행동 데이터(클릭, 조회 이력, 탐색 패턴)를 수집하거나 분석하는가?
- [ ] 이 기능이 위치정보, 생체정보, 건강정보 등 민감정보를 직·간접적으로 수집하는가?
- [ ] 이 기능이 미성년자 식별 또는 미성년자에게 특정 콘텐츠를 노출하는가?

### 외부 서비스 연동 관련

- [ ] 이 기능이 유저 데이터를 현재 처리방침에 기재되지 않은 외부 서비스(API, 파트너사)로 전송하는가?
- [ ] 이 기능이 제3자 SDK나 스크립트를 추가하는가? (광고 트래커, 분석 도구 등)
- [ ] 이 기능이 카카오/Google 외 새로운 외부 API를 사용하는가?

### 저작권 관련

- [ ] 이 기능이 책 문장을 기존보다 더 많은 양, 또는 더 자동화된 방식으로 수집·저장하는가?
- [ ] 이 기능이 책 표지 이미지, 목차, 작가 프로필 등 저작물을 서비스 내에 재저장하거나 변형하는가?
- [ ] 이 기능의 결과물이 원저작물을 대체하거나, 사용자가 책을 구매하지 않아도 되는 경험을 제공하는가?

### 소셜·커뮤니티 관련

- [ ] 이 기능이 한 유저의 정보를 다른 유저에게 노출하는 방식을 변경하는가?
- [ ] 이 기능이 유저 간 직접 메시지, 그룹, 모임 등 제3자 간 개인정보 공유를 발생시키는가?
- [ ] 이 기능이 광고성 정보(마케팅 이메일, 푸시)를 발송하는가?

### 유료화 관련

- [ ] 이 기능이 결제, 구독, 포인트, 환불 등 금전 거래를 발생시키는가?
- [ ] 이 기능이 특정 콘텐츠에 대한 접근을 유료로 제한하는가?

## 현재 법적 구조에서 기획 금지/주의 기능 목록

### 즉시 구현 금지 (현재 법적 준비 미완료 상태에서)

| 기능 | 금지 이유 | 해제 조건 |
|------|-----------|-----------|
| 마케팅 이메일/푸시 발송 | 정보통신망법 제50조 — 광고성 정보 수신 동의 체계 미구축 | 동의 수집 DB 설계 + 처리방침 업데이트 + 수신거부 메커니즘 구현 후 |
| 유료 구독/결제 기능 | 전자상거래법 준수 체계 미구축, 이용약관 미게시 상태 | 이용약관 게시 + 환불 정책 수립 + 결제 대행사 계약 후 |
| 독서 모임/그룹 기능 | 제3자 간 개인정보 공유, 만 14세 미만 동의 체계 없음 | 별도 Legal 검토 + 처리방침 업데이트 후 |
| 외부 광고 플랫폼 SDK 삽입 | 유저 동의 없는 제3자 트래킹 — PIPA 위반 | 처리방침에 광고 트래커 기재 + 쿠키 동의 배너 구현 후 |

### 높은 리스크로 주의 구현 필요

| 기능 | 리스크 | 구현 시 필수 조치 |
|------|--------|------------------|
| 책 요약/전문 인용 기능 | 책 내용 대량 복제 — 저작권법 복제권 침해 직결 | OCR로 추출된 전체 텍스트를 저장하는 구조 금지. 반드시 밑줄 감지 문장만 저장하는 현재 아키텍처 유지 |
| "이 책 전체 밑줄 모음" 페이지 | 동일 책 집적 = 사실상 책 내용 재구성 리스크 | 동일 책 전체 밑줄 목록 노출 시 반드시 출처 표시 강화 + 저작권 침해 신고 채널 연결 |
| 유저 독서 이력 기반 프로파일 공개 | 어떤 책을 읽는지가 민감정보 간접 노출 가능 | 프로파일 공개 범위 유저 제어 가능하도록 설계 |
| SNS 자동 연동 공유 (OAuth 기반 자동 포스팅) | 유저 SNS 계정 접근 = 새로운 개인정보 처리 | 반드시 수동 공유(다운로드 후 유저가 직접 업로드)로 설계. 자동 포스팅 구조 금지 |
| 사용자 얼굴 포함 사진 처리 | 생체정보(개인정보보호법 제23조 민감정보) 해당 가능 | 현재 서비스는 책 페이지 OCR 목적. 얼굴 감지 기능 추가 시 별도 동의 체계 필수 |
| 알림 기반 재방문 유도 (마케팅 성격) | 광고성 정보로 분류될 경우 별도 수신 동의 필요 | 복습 알림은 서비스 알림으로 설계(비광고), 새 책 추천 알림은 별도 동의 체계 병행 |

### Planner 판단 지침 — 기획 안티패턴과 법적 리스크의 교차점

| 기획 안티패턴 (planner-policy.md) | 법적 리스크 교차점 | 통합 판단 |
|-----------------------------------|--------------------|-----------|
| 독서량 게이미피케이션 (완독 뱃지, 스트릭) | 독서 이력 집적 = 프로파일링 데이터 증가, 처리방침 업데이트 필요 | 기획 이유 + 법적 이유 모두 기각. P2 이후에도 Legal 검토 필수 |
| 조기 추천 알고리즘 | 프로파일링 규제, 자동화된 결정 고지 의무 | 기획 기준(MAU 10,000명)과 법적 준비(처리방침 업데이트, 개인화 거부권 구현) 동시 충족 후 진행 |
| 복잡한 태그/분류 시스템 | 추가 개인정보 수집 항목 발생 가능 | 태그가 취향·성향 정보로 해석될 경우 민감정보에 준하는 관리 필요. 현재 허용 범위(온보딩 5개 취향 태그)는 Low Risk 유지 |
| DM/그룹채팅 | 제3자 간 개인정보 공유, 메시지 보존 의무 | 서비스 범위 밖 + 법적 리스크 모두 높음. 영구 기각 권장 |

## 저작권 리스크 완화를 위한 기획 가이드라인

OCR/밑줄 저장 기능 설계 시 아래 요건을 기획 수용 기준에 반드시 포함한다.

### 설계 원칙 1 — 밑줄 선택의 주체는 유저

OCR이 텍스트를 추출하더라도, 어떤 문장을 저장할지는 반드시 유저가 확인·선택하는 플로우를 유지한다. 서비스가 텍스트를 자동으로 대량 저장하는 구조는 복제권 침해 직결.

- **현재 플로우 (적합)**: OCR 추출 → 밑줄 감지 → 유저 확인/수정 → 저장
- **금지 구조**: OCR 추출 → 전체 텍스트 자동 저장

### 설계 원칙 2 — 출처 표시는 저장 플로우의 일부

밑줄 저장 플로우에서 책 제목·저자·출판사가 확정되지 않으면 저장이 완료되지 않는 구조를 유지한다. 출처 불명 밑줄 저장은 저작권법 제37조 위반.

- **기획 수용 기준 필수 항목**: "책 정보 없이는 밑줄 저장 불가 (책 정보 미확인 시 저장 버튼 비활성화)"
- **카카오 API에서 책 정보 자동 수집 실패 시**: 수동 입력 폼 제공 필수

### 설계 원칙 3 — 안전한 문장 저장 범위 준수

| 기준 | 기획 시 UI/UX 반영 방법 |
|------|------------------------|
| 200자 이내 권장 (1~2문장) | 밑줄 편집 UI에서 선택 가능한 텍스트 길이 소프트 제한 표시 (경고, 강제 차단 아님) |
| 1페이지 이내 | 사진 한 장에서 감지되는 밑줄만 저장 — 여러 페이지 연속 업로드 기능 기획 시 Legal 검토 필수 |
| 동일 책 집적 모니터링 | 동일 책 동일 유저 월 30개 이상 저장 시 어드민 모니터링 플래그 — 기획 단계에서 어드민 대시보드 설계 시 포함 |

### 설계 원칙 4 — 공유 카드의 저작물 비중 통제

공유 카드 테마 설계 시 책 문장이 카드의 유일한 주인공이 되는 구조를 피한다. 현재 6가지 테마는 이 원칙을 충족한다.

- **카드 필수 표시 요소**: 책 제목 + 저자 (모든 테마 공통)
- **권장 추가 표시**: 출판사, ISBN (카드 여백 허용 시)
- **금지 구조**: 책 문장만 단독으로 표시하고 출처 없는 카드 생성

### 설계 원칙 5 — 저작권 침해 신고 → 기획에서 인앱 신고 동선 설계

저작권 침해 신고 채널은 이메일만으로는 부족하다. 향후 기능 기획 시 인앱 신고 버튼 동선을 포함한다.

- **단기 (즉시)**: 이용약관에 신고 이메일 명시
- **중기 (P1)**: 밑줄 카드 상세 페이지 (...) 메뉴에 "저작권 신고" 항목 추가
- **수용 기준 포함 예시**: "밑줄 상세 (...) 메뉴에 신고 항목이 표시되며, 탭 시 사유 선택 + 제출 플로우로 이동한다"
